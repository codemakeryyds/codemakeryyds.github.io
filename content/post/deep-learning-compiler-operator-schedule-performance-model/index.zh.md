前面看 MLC 的时候，我一开始容易把“算子”和“算子实现”混在一起。

比如看到 `MatMul`，直觉上会觉得它就是矩阵乘法；看到 `Conv2D`，就觉得它就是卷积。这个理解只能说明模型要算什么，但离真正部署还差一大截。

真正影响速度的地方是：这个算子在目标硬件上到底怎么跑。

同样一个矩阵乘法，在 CPU 上要考虑 cache、SIMD、多线程；在 GPU 上要考虑 thread block、warp、shared memory、tensor core；如果放到端侧 NPU，还要看它支持哪些 kernel、buffer 多大、数据布局能不能对上。

这篇不是系统教程，更像我给自己补的一层理解：**算子定义语义，调度改执行方式，性能模型帮编译器做选择。**

## 算子只回答“算什么”

深度学习模型最后会落到一堆算子上：

```text
Conv2D
MatMul / GEMM
BatchNorm
ReLU
Softmax
LayerNorm
Add
Transpose
```

这些名字告诉我们计算语义，但不告诉我们实现细节。

拿矩阵乘法来说，最朴素的三层循环当然能算对：

```c
for (int i = 0; i < M; i++) {
    for (int j = 0; j < N; j++) {
        for (int k = 0; k < K; k++) {
            C[i][j] += A[i][k] * B[k][j];
        }
    }
}
```

但这段代码只是在表达数学关系。它没有照顾 cache，没有向量化，没有并行，也没有把数据复用设计进去。

所以我现在更倾向于把算子看成“语义接口”：它告诉编译器要做哪类计算，但具体怎么铺到硬件上，是调度要解决的问题。

## 调度是在改“怎么算”

调度源语就是编译器改变执行方式的一组工具。它们不会改变结果，但会改变循环结构、内存访问和并行方式。

常见源语大概是这些：

| 源语 | 作用 | 我自己的理解 |
|---|---|---|
| `split` / `tile` | 拆循环、分块 | 不要一口吃完整个矩阵，切小块提高复用 |
| `reorder` | 调整循环顺序 | 让内存访问更顺，减少 cache miss |
| `vectorize` | 向量化 | 一条 SIMD 指令处理多个元素 |
| `unroll` | 展开循环 | 少一点循环控制，多一点指令级并行 |
| `parallel` | 并行化 | 把不同块分给不同线程或计算单元 |
| `cache_read` / `cache_write` | 缓存读写 | 把热点数据放进更近的存储 |
| `compute_at` | 控制中间计算位置 | 决定中间结果在哪层循环里算 |
| `inline` | 内联简单计算 | 减少多余中间结果 |

这些词看起来很“编译器”，但核心其实很朴素：**让数据少搬一点，让计算单元多忙一点。**

比如对 `MatMul` 做 tile，可能会变成这种形态：

```c
for (int io = 0; io < M; io += 32) {
    for (int jo = 0; jo < N; jo += 32) {
        for (int i = io; i < io + 32; i++) {
            for (int j = jo; j < jo + 32; j++) {
                for (int k = 0; k < K; k++) {
                    C[i][j] += A[i][k] * B[k][j];
                }
            }
        }
    }
}
```

这个例子还很粗糙。真实系统里还会继续调循环顺序、向量宽度、展开因子、线程绑定、shared memory 使用方式。也正因为组合太多，人手工挑最优方案很难。

## 调度空间为什么会爆炸

一个矩阵乘法看起来简单，但可以选的参数很多：

- tile size 取多少；
- `i/j/k` 三层循环怎么排序；
- SIMD 宽度怎么用；
- 哪一层循环展开；
- 哪一层并行；
- A、B、C 哪些数据要进 cache；
- 中间结果什么时候写回；
- 数据布局要不要变。

这些选择组合起来，很快就不是“试几个版本”能解决的了。

所以 TVM AutoTVM / Ansor、Halide Auto-scheduler、Triton autotune 这类系统才有意义。它们不一定用同一种方法，但都在回答一个类似问题：

> 这个计算，在这块硬件上，哪种执行方式更划算？

这里的“划算”也不只是快。编译时间、内存峰值、部署体积、动态 shape 支持，都会影响最后选择。

## 性能模型不能只看 FLOPs

以前我看算子性能，很容易先看计算量。比如一个大 `MatMul` 有多少 MACs，一个卷积有多少 FLOPs。

这个指标有用，但只看它会误判。

一个粗暴估计是：

```text
执行时间 ≈ 计算量 / 硬件峰值算力
```

问题是，真实执行经常卡在别的地方：

- 内存带宽；
- cache 命中率；
- 数据复用；
- SIMD 利用率；
- kernel launch overhead；
- layout conversion；
- 中间结果读写；
- 算子是否能融合；
- 硬件有没有对应特殊指令。

所以性能建模要看的不是“这个算子快不快”，而是这一组东西：

```text
operator + shape + dtype + layout + schedule + hardware
```

少一个维度，结论都可能变。

## Roofline 对我最有用的一点

Roofline Model 里有一个概念叫算术强度：

```text
Arithmetic Intensity = 计算量 / 数据搬运量
```

我觉得它最有用的地方，是帮我区分两类瓶颈。

如果一个算子每搬一次数据能做很多计算，它更可能是 `compute-bound`。大规模 `MatMul` 就比较典型。它的计算量约是：

```text
M × N × K 次 MAC
```

如果按 FLOPs 统计，通常约是：

```text
2 × M × N × K
```

而 A、B 的元素会被反复复用，所以大矩阵乘法比较容易把计算单元喂饱。这也是 GPU、NPU、TensorCore 都很重视 GEMM 的原因。

反过来，`ReLU`、`Add` 这类算子计算很轻。以 ReLU 为例：

```text
y = max(x, 0)
```

每个元素只做一次比较，但输入要读，输出要写。它的瓶颈很容易变成访存。

这时候单独优化 ReLU 的计算指令意义有限。更常见的做法是融合。

## 融合的价值在于少写中间结果

比如：

```text
Conv → BatchNorm → ReLU
```

如果三个算子分开跑，中间结果会反复写回内存、再读出来。

推理阶段可以先做 BN folding：

```text
BN(Conv(x, W, b)) → Conv(x, W', b')
```

然后再把 `Conv + ReLU` 融成一个 kernel。这样 Conv 算完以后直接做 ReLU，最终结果写回一次。

这里的重点不是“融合听起来高级”，而是它减少了内存流量。对于 memory-bound 算子，这往往比抠几条计算指令更有用。

## 性能模型大概有三种层次

工程里不会所有东西都靠自动搜索。不同系统会混合使用几类方法。

**第一类是手工规则。**

比如大规模 MatMul 走某个加速库，小算子回退 CPU，`ReLU/Add` 尽量融合，`Reshape` 尽量消除。这类规则可控、容易落地，但维护成本高，对新 shape 和新硬件不一定泛化。

**第二类是解析代价模型。**

它会估计计算时间和访存时间：

```text
Compute_Time = Ops / Effective_Compute_Throughput
Memory_Time = Bytes / Effective_Memory_Bandwidth
Time ≈ max(Compute_Time, Memory_Time) + Overhead
```

这个模型不精确，但能告诉你优化方向。比如某个卷积算出来访存时间明显大于计算时间，那就应该优先想 layout、融合、复用，而不是只盯着算力。

**第三类是实测加搜索。**

系统生成多个候选 schedule，编译后在目标硬件上跑，记录时间，再继续搜索。AutoTVM、Ansor、Triton autotune 都属于这个方向。

它的好处是能适应真实硬件，坏处也明显：搜索慢，系统复杂，还需要目标硬件反馈。

## 编译器最后是在做选择题

把这些东西放一起看，深度学习编译器其实一直在做选择：

```text
这个算子放 CPU、GPU 还是 NPU？
这个 shape 用哪个 kernel？
这个 layout 要不要转换？
这个中间结果要不要写回？
这个算子能不能融合？
这个 schedule 值不值得编译？
```

所以它不是简单地把 ONNX、TensorFlow 或 PyTorch 模型换个格式。格式转换只是入口，后面还有图优化、算子选择、调度搜索、性能建模、内存规划和代码生成。

## 我现在的理解

这篇写完以后，我对“机器学习编译”的理解更具体了一点。

算子是语义层，它说清楚要算什么。

调度是实现层，它决定循环、并行、缓存和数据移动怎么组织。

性能模型是决策层，它判断某个实现放在某块硬件上到底值不值得。

以前我会把“优化模型部署”理解成换框架、转格式、开量化。现在看，真正难的地方在更下面：同一个计算，换一种执行组织方式，性能可能完全不一样。
